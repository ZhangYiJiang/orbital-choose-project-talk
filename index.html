Rank your priorities

Evaluate the project types

To choose your project type, consider these criteria. Rank them according to what you think is important, then see how closely the project type you're thinking of matches up.

Reach and accessibility

Who can access or use your project, and is it easy to do so?

Reach and accessibility is important if you're building you actually want people to use. It's less important if your project is a toy project or a portfolio piece. In general, the more platforms your app is on, and the less steps there are before the user can actually get productive with it - for instance, require creating an account or installing something, the more likely it is for them to actually use your app.

Educational

What do you learn from building the project?

For many of you this will be your first time building an actual app, so it's good to consider what you yourself will get out of the experience. Also consider how the project will look in your resume or portfolio. If this is not a 'real' app, you can take more liberty in choosing the newest technology, instead of something realistic.

Development experience

Is it fun or easy to build? Is it expensive?

Often overlooked, your own experience as a developer does matter. A good development experience will keep you motivated.

Competitors

What do existing apps use?

Why did they choose it, and should you try to differentiate?

Another frequently overlooked method for evaluating just about anything is look at your competitors. Your ideas are most likely original, so see what others before you have done (and if there aren't maybe you should ask yourself why that's the case. You can learn a lot from your competition - and not just to copy, but also to differentiate yourself.

Monetization

Can you make money?

Probably not your top concern, but this is an important consideration for real projects.

How websites work

Frontend

HTML/CSS/JavaScript

The code that's sent to and evaluated by the browser is known as the 'frontend'. This is the presentation of your site - the thing the user sees and interacts with.

Static website

For simple websites, where the content (HTML) does not change, this is actually all you need. An example of this is NUS Computing's FOP website. But this isn't going to work for most apps - they need to take in and store user inputs. Backend. Anything running on the server is known as the backend. In traditional web applications the server will handle the business logic, data persistence, as well as translating the data into HTML that the client will use.

Web Applications

Single Page Application

There's also a new breed of web applications that shifts more of the responsibility to the client.

Pros

Reach - Easily cross platform

The biggest advantage of building web apps is that they can be accessed from any device - desktop, mobile, smart TVs - anything with a browser, really.

Pros

Accessible - No installation required

There's also no need to install anything, which is great because the biggest barrier to users these days is installing the app, and creating an account. Web apps just removed one of the two steps.

Pros

Easy to learn - start simple, learn as you go along

This is subjective, but in my opinion, HTML and CSS at least are very easy to learn. That's all you need for a simple website. And if you want your website to do more, simply pick up more skills as you go along. Your old skills will still come in handy.

Pros

Easy to develop - No compilation, wide choice of languages

While native apps usually use compiled languages like Java and Swift, web apps are mostly written in interpreted languages. This means you can refresh your browser, and the changes made to the code are reflected immediately in your website. In addition, remember that for backends you can choose pretty much any language you're comfortable with and there will be a web framework for it.

Cons

No direct hardware access - weaker API, poor performance

The most obvious downside for web apps is the lack of native hardware access. Native refers to the actual hardware the browser your user is using - be it a desktop computer, a tablet, or a mobile phone. You're building on top of the browser, not the native platform itself.

Cons

Weaker and inconsistent API

That means what you get access to is generally a subset of what native apps have access to. For example, the HTML5 WebRTC API does not give you camera FOV, which limits their application for AR.

Cons

(Relatively) poor performance

The browser itself also adds overhead. While most apps don't need every CPU cycle their device can give, and performance has improved significantly, you still don't want to build performance sensitive apps such as games in the browser.

Cons

Non-native UI

Your UI also won't look like the native platform's. Sure, you can mimic the native UI, but that usually a lost cause - you can never match the efficiency of native UI, and some UI elements like the blur and vibrancy effect on iOS is absolutely impossible to achieve in the browser.

Cons

Fast changing - short knowledge lifespan

The web moves fast - really fast. That's good, because you get better technology, better performance, but it also means what you learned here may not be relevant in a few year's time.
